# Nexa — Security Baseline

This document states what security behavior exists **today**, in code, versus what is **planned** for the Identity phase (Phase 3 of [PROJECT_ROADMAP.md](PROJECT_ROADMAP.md)) or later. Do not treat anything in the "Planned" column as already protecting the system.

| Concern | Implemented now | Planned |
|---|---|---|
| Password hashing | `PasswordHasher` (Infrastructure) wraps `BCrypt.Net` `EnhancedHashPassword`/`EnhancedVerify` at work factor 12. Salt is embedded in the hash — no separate salt column/field anywhere. **Confirmed exercised end-to-end** by Tenant Onboarding — `TenantOnboardingService` hashes the owner's password before it ever reaches Infrastructure/SQL, and only the hash is passed to `tenant.usp_Organization_Register`. | — |
| Token hashing | `TokenHasher` (Infrastructure) generates and hashes raw tokens (refresh/confirmation/reset) via SHA-256; only the hash is ever persisted (`identity.RefreshTokens.TokenHash`, etc. — see [database blueprint](database/DATABASE_FINAL_BLUEPRINT.md)). **Confirmed exercised end-to-end** for email-confirmation tokens by Tenant Onboarding: `ITokenHasher.GenerateRawToken()`/`Hash()` run before the database call, only the hash reaches `tenant.usp_Organization_Register`/`identity.EmailConfirmationTokens`, and a unit test (`TenantOnboardingServiceTests`) asserts the raw value is never what gets persisted. | Same confirmation still owed for refresh/reset tokens once Login/Forgot-Password are implemented (Phase 3). |
| Refresh-token rotation | `AuthService.RefreshTokenAsync` issues a new token and revokes the old one on each refresh; DB schema carries `TokenFamilyId`/`ReplacedByTokenId` for reuse detection (migration 009). | Wiring the *detection* logic itself (revoke the whole family on reuse of an already-revoked token) into `AuthService`/`AuthRepository` — schema is ready, service logic is not yet confirmed wired end-to-end. Verify before relying on it. |
| Token revocation | `RefreshTokens.RevokedAt`/`RevokedBy`/`RevokedByIp` columns exist; `identity.UserSessions` (migration 009) models per-device sessions for "log out everywhere." | Actual session-revocation-cascades-to-tokens logic, and a "list my active sessions" endpoint. |
| Email confirmation | **Fully implemented and verified live**: Tenant Onboarding creates the initial token (migration 011); `POST /api/auth/confirm-email` and `POST /api/auth/resend-email-confirmation` (migration 013, `IEmailConfirmationService`) consume/reissue it. Concurrency-safe (`UPDLOCK, HOLDLOCK` — a genuine parallel double-confirm was tested live and correctly serialized), idempotent, enumeration-safe (resend returns byte-identical responses for nonexistent/confirmed/rate-limited accounts — verified live), fully audited (every attempt writes an `audit.AuditLogs` row with an internal `FailureReason` never exposed to the client). See [docs/EMAIL_CONFIRMATION.md](EMAIL_CONFIRMATION.md) for the full design. | Per-user resend cooldown/hourly-cap is enforced in SQL (config-driven, not hardcoded), but token-creation and email-enqueue are two separate round trips, not one transaction — see "Email delivery failure strategy" in the linked doc for why this matches an existing, accepted precedent (Tenant Onboarding has the same characteristic). No scheduled cleanup of expired/used/revoked tokens yet (deliberately deferred — see "Future cleanup strategy"). |
| Password reset | Equivalent hashed-token flow (`SavePasswordResetToken`, `ValidatePasswordResetToken`, `ResetPassword`) exists in `AuthService`/`AuthRepository`. | Same caveat — stored procedures not yet written. |
| Sign-in logging | `identity.SignInLogs` table exists with `EventType`, `IpAddress`, `UserAgent`, `CorrelationId` (migration 009); no FK (append-only, informational). | Application code actually writing to it on every login attempt — not yet confirmed wired in `AuthService`. |
| Lockout policy | `Users.FailedLoginAttempts`/`LockoutEndDate` columns exist with a `CHECK` constraint (`FailedLoginAttempts >= 0`); `AuthenticationOptions.MaxFailedLoginAttempts`/`LockoutDurationMinutes` are bound from config. | Confirming the increment/reset/lockout-check logic in `AuthService` matches these option values exactly (review before shipping). |
| Role-based authorization | DB model exists: `identity.Roles` (tenant-local copies of system templates), `Permissions`, `RolePermissions`, `UserRoles`, all tenant-safe-FK-enforced (migration 009). | JWT claim population from the effective permission set, and the WebApi-side authorization policy wiring (no endpoints exist yet to attach policies to). |
| Fine-grained permissions | Seed catalog exists (`identity.Permissions`, migration 008) — e.g. `Customer.View`, `Payment.Create`. | Endpoint-level `[Authorize(Policy = "...")]`-style enforcement once endpoints exist. |
| Session management | `identity.UserSessions` table exists (migration 009). | Application/Infrastructure code to create/update/revoke session rows; no service currently writes to this table. |
| Audit logging | `audit.AuditLogs` table exists with JSON-validity `CHECK` constraints on before/after values (migration 009). | Nothing currently writes to it — no audit-writing code exists yet in Application/Infrastructure. This is explicit Phase 1 roadmap work, not done. |
| Secret management | `appsettings.json` ships only empty-string placeholders for `Jwt:SecretKey`, `ConnectionStrings:SqlConnection`, and `Smtp:*` credentials — nothing real is committed. | Real secrets belong in user-secrets locally and a proper secret store (Key Vault or equivalent) in deployed environments — not yet configured, since there's no deployment target yet. |
| Input validation | FluentValidation validators exist for every Authentication request DTO (`RegisterValidator`, `LoginValidator`, etc.), auto-registered via `AddValidatorsFromAssemblyContaining`. | Validators for every future feature's request DTOs, per [DEVELOPMENT_WORKFLOW.md](DEVELOPMENT_WORKFLOW.md)'s checklist. |
| Rate limiting | **Implemented for the one public endpoint that exists**: `POST /api/organizations/register` is limited to 5 requests/minute per IP via ASP.NET Core's built-in fixed-window rate limiter (`WebApi/Program.cs`, policy `RateLimiterPolicies.PublicRegistration`). Rejections return a real HTTP 429 (not wrapped in the `ApiResponse` 200 convention, since this is a transport-level protection, not a business outcome). | Apply the same pattern to login/password-reset endpoints once they exist (Phase 3). No CAPTCHA or device fingerprinting yet — see [docs/TENANT_ONBOARDING.md](TENANT_ONBOARDING.md) "Current limitations". |
| CORS | Not configured (no `AddCors`/`UseCors` in `Program.cs` yet). | Configure explicitly once a frontend origin is known — never `AllowAnyOrigin()` combined with credentials. |
| HTTPS | Kestrel default HTTPS redirection is not yet explicitly configured in `Program.cs`. | Add `UseHttpsRedirection()` and HSTS before any non-local deployment. |
| Secure headers | Not implemented. | Standard security headers (X-Content-Type-Options, X-Frame-Options/frame-ancestors, Referrer-Policy) before any non-local deployment. |
| Centralized logging | **Implemented**: Serilog (`builder.Host.UseSerilog()`) writes to Console and to a dedicated `NexaLogs` SQL Server database (`Serilog:WriteTo:MSSqlServer`, table `dbo.ApplicationLogs`, auto-created) — separate from the application's own `Nexa` database, so log volume/retention never competes with transactional data. Every request is enriched (`CorrelationIdMiddleware`, `RequestLoggingMiddleware`) with `CorrelationId`, `UserId`, `OrganizationId`, `IPAddress`, `RequestPath`, `RequestMethod` as first-class queryable columns, not buried in an XML blob. Verified end-to-end: a real registration request's log rows and its `audit.AuditLogs` rows carry the identical `CorrelationId`. | `RequestBody` capture is Development-only and redacts registration/login/password endpoints by path match (`RequestLoggingMiddleware.CaptureBodyAsync`) — this is a path-substring allowlist, not a schema-aware redactor; review it whenever a new sensitive endpoint is added. |
| Sensitive-data logging restrictions | No logging call anywhere currently logs a raw password, raw token, or full JWT (verified during cleanup — see [ARCHITECTURE_RULES.md](ARCHITECTURE_RULES.md) rule 16). **Re-verified for Tenant Onboarding**: `TenantOnboardingService`'s log statements reference only `OrganizationId`, `Slug`, and (via Serilog's `LogContext`, not the message text) the correlation id — never the password, password hash, raw confirmation token, or token hash. **Re-verified for the email template system**: `EmailTemplateService`/`SmtpEmailService`/the job handlers log recipient, template key, and job id only — never the rendered HTML, a confirmation/reset/invite link, or any placeholder value (see [docs/EMAIL_TEMPLATES.md](EMAIL_TEMPLATES.md) "Security rules"). | Keep enforcing this in review as new logging is added; consider a Serilog destructuring policy that redacts known-sensitive property names as a backstop. |
| SQL injection prevention | Every DB call goes through Dapper's parameterized `DynamicParameters` against stored procedures — no string-concatenated SQL exists anywhere in the codebase (verified during cleanup). | Keep enforcing via [ARCHITECTURE_RULES.md](ARCHITECTURE_RULES.md) rule 7/13 in review — no exceptions for "just this one dynamic filter." |
| Stored procedure parameterization | Same as above — parameters are always bound via `DynamicParameters.Add(name, value, DbType)`, never interpolated into the procedure name or a query string. | — |
| Tenant isolation | See [MULTI_TENANCY.md](MULTI_TENANCY.md) — application-layer filtering convention plus composite tenant-safe foreign keys (migration 009) are in place at the schema level. | SQL Server Row-Level Security as the third defense layer (blueprint §7); JWT `org_id` claim population/validation end-to-end (no login endpoint exists yet to issue it). |
| Static file serving | `app.UseStaticFiles()` (`WebApi/Program.cs`) serves all of `wwwroot/` publicly and anonymously, which is what makes uploaded logos/profile pictures reachable at the URLs `IStorageUtility` builds. This also means `wwwroot/resources/system-messages.json` and `system-labels.json` are publicly downloadable — low risk today (UI strings only, no secrets), but worth knowing before anything more sensitive is ever placed under `wwwroot`. | Nothing under `wwwroot/uploads` is access-controlled by folder — anyone with a file's URL (a GUID-named path) can fetch it. Fine for logos (meant to be public-facing brand assets); would need `IFileLinkService`'s signed-URL mechanism instead for anything private (e.g. receipts) before that data goes through this same pipeline. |
| File uploads | Per-file-type allow-lists via `UploadPolicies` (`Application/Common/Upload/UploadPolicy.cs`) — extension + MIME type + size cap, enforced by FluentValidation before the file reaches storage (5 MB, JPEG/PNG for profile pictures; 5 MB, JPEG/PNG/WebP for organization logos — no SVG, an XSS vector if ever served inline). `LocalFileService` path-traversal guard rejects any storage key that resolves outside the uploads root. Writes are atomic (temp file + rename) so a failed/cancelled upload never leaves a corrupt file. **Verified live** for the organization-registration logo upload, including a 50 MB non-image file reaching validation (not a transport-level rejection) and being correctly turned away by format checking — see [docs/TENANT_ONBOARDING.md](TENANT_ONBOARDING.md) "Logo upload". | The platform-wide request-body ceiling (`Storage:MaxUploadSizeBytes` via Kestrel + `FormOptions`) is **unlimited by default** — deliberate, so the storage layer is ready for legitimately large files, but it means nothing caps raw request size for an unauthenticated route beyond the existing 5 req/min rate limit and per-file-type validation. A malicious actor could still push several arbitrarily large requests/minute at the public registration endpoint before format/size validation (which only runs after the body is received) rejects them. Tighter *per-route* enforcement ahead of model binding is not yet built; not certain minimal APIs honor `IRequestSizeLimitMetadata` the way MVC does, so this wasn't implemented without verifying it actually works. If this endpoint sees real-world abuse, set `Storage:MaxUploadSizeBytes` to a concrete ceiling. |

## What "implemented now" means precisely

It means the described mechanism exists in the current codebase (a class, a table, a constraint) and was verified during this cleanup pass. It does **not** mean there is a working, end-to-end, tested feature reachable from an HTTP endpoint — there are currently **no real API endpoints** in `WebApi` (Phase 0 of the roadmap is solution cleanup, not feature delivery). Treat every "Implemented now" row as "the building block exists and compiles," and re-verify it against real behavior before relying on it in a security review of an actual running feature.

## What "planned" means precisely

It means the design exists (in this document, the database blueprint, or the roadmap) but no code currently implements it. Do not describe a "planned" item as done in any customer-facing or compliance-facing material.
